## Architecture Overview
- This is a Rust terminal emulator and agentic development environment with a custom UI framework called WarpUI.
- There are two front-ends sharing core model/UI plumbing:
  - GUI desktop app in `app/`, using `warpui` and `crates/warpui_core` GPU/pixel rendering.
  - Headless TUI in `crates/warp_tui`, using cell-grid `TuiElement` rendering in `crates/warpui_core/src/elements/tui`.
- Shared patterns include the global `App` entity store, `ViewHandle<T>` references, `AppContext`/`ViewContext` access, and action dispatch.
- Main app feature areas include terminal emulation, AI/Agent Mode, Warp Drive/cloud objects, auth, settings, workspace/session management, billing, and server APIs.

## Important Flows
- Contribution flow starts from GitHub Issues. Features generally need readiness labels and specs under `specs/`; bug fixes can proceed once actionable. See `CONTRIBUTING.md`.
- Local setup and skill restoration are handled by `./script/bootstrap`.
- GUI app run path: `./script/run` or `cargo run`.
- TUI run path: `./script/run-tui`.
- Presubmit path: `./script/presubmit`, which runs formatting checks, inline-test-module checks, clippy, clang-format, WGSL formatting, optional PowerShell lint, nextest, and doc tests.
- Integration tests use the custom framework in `crates/integration/`; see `crates/integration/tests/INTEGRATION_TESTING.md`.
- Database migrations live under `crates/persistence/migrations/`; schema generation guidance is in `app/src/persistence/README.md`.
- GraphQL schema/client surfaces include `crates/warp_graphql_schema/`, `crates/graphql/`, `app/src/server/graphql/`, and `crates/warp_server_client/`.
- Environment variable collections are documented in `app/src/env_vars/README.md`.

## Development Guidelines
- This is a Cargo workspace with 60+ member crates. Main binary code is in `app/`; custom UI framework code is in `crates/warpui/` and `crates/warpui_core/`.
- Prefer imports over long Rust path qualifiers. Keep imports at the top of the file except for cfg-guarded one-offs.
- Avoid unnecessary type annotations, especially in closure parameters.
- If a function takes `AppContext`, `ViewContext`, or `ModelContext`, name it `ctx` and place it last unless a closure parameter must be last.
- Remove unused parameters completely instead of prefixing them with `_`.
- Prefer inline format arguments such as `format!("{message}")`.
- Do not pass `Itertools::format` results directly to logging macros; use a reusable `String` such as `iter.join(", ")`.
- Avoid wildcard `_` arms in match statements when exhaustive matching is practical.
- Do not remove existing comments when making unrelated changes.
- Unit tests should live in separate `*_tests.rs` or `mod_test.rs` files and be included with a `#[cfg(test)]` module at the end of the corresponding module.
- When adding a toggleable setting, add the matching Command Palette enable/disable entry and any required context flags.

## Feature Flags
- Runtime feature checks live in `warp_core/src/features.rs` through the `FeatureFlag` enum.
- Prefer `FeatureFlag::YourFlag.is_enabled()` over compile-time cfg directives unless code cannot compile without cfg gating.
- Keep flags high-level and product-focused.
- Hide UI entry points behind the same flag that gates the behavior.
- Remove the flag and dead branches after rollout stabilizes.

## PR Workflow
- Before opening or updating a PR, run `./script/format` and the clippy/test checks in `./script/presubmit`.
- Use `.github/pull_request_template.md`.
- Add changelog entries when appropriate using the template's `CHANGELOG-NEW-FEATURE:`, `CHANGELOG-IMPROVEMENT:`, `CHANGELOG-BUG-FIX:`, and `CHANGELOG-IMAGE:` prefixes.
- Do not create public PRs or issues that disclose non-public security vulnerabilities; follow `SECURITY.md`.
