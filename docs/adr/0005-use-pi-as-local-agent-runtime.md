# Use Pi as the Local Agent Runtime

Local-only Mode will delegate model interaction, context evolution, and the agent loop to Pi through a small bridge interface. Warp remains the Tool Execution Authority and owns permission decisions, tool effects, and the user interface; Pi may request tool use but must not execute Warp tools directly. This avoids maintaining a second agent runtime in Rust without bypassing Warp's existing safety and interaction model.

## Considered Options

Continuing to expand Warp's direct-provider implementation was rejected because it duplicates Pi's agent loop, provider, context, and session capabilities. Allowing Pi to execute its built-in tools directly was rejected because Pi runs with the launching process's permissions and would bypass Warp's permission and action model.

## Consequences

Warp's Conversation Record is the canonical user-visible history. Pi may retain a Runtime Checkpoint for efficient continuation, but that checkpoint is subordinate: when the two disagree, Warp wins, and missing or invalid Pi state must be rebuilt without replacing or rewriting Warp history.

Each active Conversation Record has one dedicated Pi Runtime process. The process survives individual requests, is never shared across conversations, and may be stopped when its conversation becomes inactive; restarting it rebuilds runtime state from the Conversation Record. This isolates working directories, model state, cancellation, and process failures without paying process startup cost on every turn.

Warp is the sole owner of the Tool Catalog. Pi's built-in tools are disabled, and the bridge exposes only proxy tools described by Warp; Pi Skills may request catalog tools but cannot introduce side-effecting tools outside the catalog. Every tool request returns to Warp for permission and execution before its result is resumed in Pi.

The first Tool Catalog intentionally matches the narrow tool surface already supported by Warp's Local-only direct-provider path: `run_shell_command`, `read_files`, `apply_file_diffs`, and MCP tools that Warp has already configured. Search and indexing, documents and artifacts, computer use, orchestration, subagents, and cloud- or server-dependent tools remain out of scope. A later tool family is added only after its input schema, Tool Result Projection, permission behavior, cancellation, and recovery semantics are explicitly supported by the Bridge Protocol and Tool Execution Authority.

Warp assigns both an opaque stable identity and a Provider-visible function name to each Tool Catalog entry; the Bridge never renames tools. Built-ins keep their existing names. MCP tools keep Warp's current deterministic, 64-character-bounded encoding derived from the server installation UUID and raw tool name, including its collision-resistant suffix. Pi sends the Provider-visible name verbatim and maps a returned name back through the immutable catalog snapshot; an unmapped name is an Invalid Tool Request.

Warp is also the sole owner of Provider configuration and API Keys. The bridge receives only the runtime values needed by its dedicated child process and must not read Pi credential or model files, persist secrets, or expose them through protocol events, logs, or errors.

The Provider Origin is fixed from Warp's configured Base URL. A Provider request may follow at most three redirects, and only when the destination has the same scheme, host, and effective port as that origin. A cross-origin, cross-port, or HTTP/HTTPS redirect fails with the redacted `provider_redirect_not_allowed` code, and the API Key is never sent to the redirected destination. HTTP Base URLs remain supported under ADR-0002 with their existing accepted transport risk; this redirect restriction does not represent HTTP as secure.

The first increment supports only Warp's existing OpenAI-compatible Provider configuration. Pi's additional Provider integrations remain out of scope so replacing the Agent Runtime does not also expand credential, OAuth, model-capability, error-mapping, and validation behavior.

Pi's underlying `openai-completions` SDK retries remain disabled with `maxRetries: 0`, while the Agent Runtime may make one additional Provider Attempt for a recoverable transport error, HTTP 408, HTTP 429, HTTP 5xx response, or unexpected EOF. This retry is allowed only before that Provider Attempt has emitted any text, tool-call data, or other model output. It keeps the same `run_id`, increments an internal attempt counter, and creates no Conversation Record message. Once any model output has been emitted, failure terminates the Agent Run; Pi and Warp do not automatically resume it. This preserves Warp's safe pre-output Local-only retry without allowing an opaque SDK retry or replay after visible output.

An Agent Run remains owned by Pi across model turns and tool execution. When Pi emits a tool request, the run waits while Warp applies permission policy and executes the tool; Warp then returns the tool result to that same run, which decides whether another model turn is needed. Warp must not split tool follow-up into a new Agent request or reconstruct the agent loop itself.

A Tool Denial is a structured tool result, not Agent Run cancellation. Pi receives whether the user or policy denied execution and may adapt, explain the limitation, or finish; only an explicit run cancellation or terminal runtime/protocol failure stops the run for that reason.

An Invalid Tool Request with a stable, unique `tool_call_id` also becomes a persisted Tool Execution Record with an `error` Tool Result Projection. Warp does not request permission or execute an effect, Pi may correct the request within the same Agent Run, and the request counts toward the run's tool-call limit. A missing or duplicate `tool_call_id`, or an undecodable tool-call frame, instead terminates the run with the redacted `provider_protocol_error` code because Warp cannot correlate recovery delivery or guarantee idempotency. This deliberately improves the Pi path over the Rust Local Provider's current fail-the-request behavior without changing that legacy runtime.

Warp supplies a Tool Request Limit in Run Configuration, initially preserving the Local Provider's default of 32 requests per Agent Run. Pi must not begin another model turn after the limit is reached. If one model response emits a batch that crosses the limit, Warp executes only the requests within the remaining budget and returns persisted `tool_limit_reached` error outcomes for the rest. The run then ends with the distinct `limit_reached` terminal outcome. Warp may present a confirmation action, but confirmation creates a visible user input and a new Agent Run; the first Bridge Protocol does not pause or resume the exhausted run.

Agent Run Cancellation first stops any pending Warp tool approval or execution and asks Pi to abort the active model stream and loop. The dedicated Pi process and its recoverable state remain available for the next user input. Warp terminates and later rebuilds the process only when it does not acknowledge cancellation within a bounded grace period.

If Pi exits during streaming, Warp retains already displayed text as Interrupted Output in the Conversation Record instead of deleting it. Reconstruction starts from the last completed message or tool result and excludes the interrupted assistant message as completed model context. Warp may offer an explicit retry but must not silently replay a run that may already have produced tool effects.

An explicit Retry action creates a Retry Run with a new `run_id`, a `retry_of_run_id` reference, and the latest committed Runtime Transcript revision. It does not duplicate the original user message or inject a hidden retry prompt. Interrupted Output remains visible but excluded from model context, while committed tool requests and outcomes remain in the transcript and may be redelivered without re-executing their effects. This differs from continuing after `limit_reached`, which requires a new visible user input because the prior run exhausted its allowed work rather than failing to deliver the requested work.

Warp owns the Agent Resource Catalog for the first increment. Pi receives only the Skill instructions and project context selected by Warp, does not scan Pi or project resource directories, and starts with Pi Extensions disabled. This prevents a second discovery and trust model from introducing hidden instructions, startup code, or tools outside Warp's catalogs.

The first Agent Resource Catalog contains only the global and project rules that Warp resolves for the run's working directory, context explicitly attached by the user for that input, and the complete instructions of a Skill explicitly invoked by the user. It does not advertise every installed Skill, let Pi select a Skill automatically, or expose `read_skill`. Broad Skill discovery is deferred until a bounded catalog and resource-loading contract can be added without eagerly injecting every Skill into model context.

Warp materializes those selected resources as a bounded Resource Snapshot attached to the input that initiated the Agent Run. The Conversation Record retains the exact normalized contents, and the Runtime Transcript projects them when rebuilding Pi context. Historical rules, attachments, and Skill instructions are never reconstructed by rereading their current source files; changed sources affect only Resource Snapshots created for later runs. This extends Warp's existing persisted input-context behavior instead of introducing a second Bridge-owned resource store.

The Bridge Protocol begins with an explicit version and capability handshake. A Core Protocol major-version or schema-hash mismatch, or a missing required capability, fails closed with an actionable error. Warp sends no API Key, Conversation Record content, Tool Catalog, or Agent Resource Catalog before the handshake succeeds, and neither side silently ignores unknown required messages.

The authoritative Protocol Schema lives in Warp's `tools/warp-bridge/protocol/` directory as versioned JSON Schema plus valid and invalid JSONL conformance fixtures. Bridge TypeScript and Warp Rust retain native strict types in the first increment rather than introducing cross-language code generation, but both validate protocol boundaries, reject fields outside the negotiated schema, and run the same fixtures against the pinned Bridge binary. The handshake includes schema identity in addition to protocol version and capabilities so a stale or drifted implementation fails before receiving secrets or conversation content.

The Core Protocol has only a major version and `core_schema_hash`; it contains the envelope, handshake, Transcript Sync, Agent Run lifecycle, cancellation, and events required by the first increment. A Core change that alters its schema increments the major version and requires Warp to bundle a matching Bridge binary. Optional evolution occurs through Protocol Capabilities such as `usage.v1` or `parallel_tools.v1`, each with its own schema hash. A capability is enabled only when its name, version, and hash match on both sides; otherwise it is unavailable, and a required mismatch fails the handshake.

Bridge source lives in Warp's dedicated `tools/warp-bridge` package. Warp vendors exact upstream Pi release packages under `third_party/pi`, records their source revision and checksums, and preserves their licenses; vendored Pi code supplies the Agent Runtime implementation but never becomes the Bridge Protocol source of truth. Warp builds a standalone, non-TUI Bridge binary for each supported platform, bundles that binary as an application resource, and owns launch and supervision. The integration does not use the experimental Pi orchestrator protocol and does not depend on a user-installed Node, Bun, or Pi runtime.

The initial artifact matrix is the intersection already shipped by both repositories: macOS arm64 and x86_64, Linux arm64 and x86_64, and Windows arm64 and x86_64. Release builds require the matching Bridge artifact for their target and fail rather than producing an application that advertises Pi support without its runtime. Each target receives the same Core Protocol conformance and startup-handshake smoke test before packaging.

Warp CI builds `warp-bridge`-only artifacts for those six targets from the checked-in Bridge source, locked dependencies, and verified vendored Pi packages rather than bundling the full Pi CLI distribution. Warp checks in a Bridge Artifact Manifest containing the exact Bridge version, vendored Pi version and source revision, Core Protocol schema hash, and expected SHA-256 digest and byte size for each target. Build tooling verifies the selected artifact, caches only by its verified digest, and fails on absence or mismatch. The application bundle contains only its target binary and subjects it to the platform's normal signing or notarization flow; the installed application never downloads or updates the Bridge at runtime. Development builds may use an explicit local Bridge path, but release builds reject that override.

The bundled Bridge binary is trusted application code in the first increment, not an adversarial process. Warp still launches it with a minimal inherited environment and private runtime working directory, while Pi tools, Extensions, and resource discovery remain disabled. Full cross-platform OS sandboxing is deferred, and the product must not claim that this design contains a compromised Bridge binary.

Every Agent Run and tool request has a stable identity. Warp persists a Tool Execution Record before delivering its outcome to Pi, and Pi acknowledges or idempotently accepts repeated delivery. Recovery may resend a recorded outcome but must never re-execute the same tool-call identity, including when Pi exits after the effect but before acknowledging the result.

Tool-call identity is scoped by the composite `(conversation_id, run_id, tool_call_id)`. A `tool_call_id` must be unique within one Agent Run but need not be globally unique across Provider responses or later runs. Reusing an identity inside a run is a protocol failure unless the request is byte-for-byte the already recorded request being redelivered during recovery.

Runtime Binding and the current Conversation Record Revision are stored as optional, backward-compatible fields in the existing `agent_conversations.conversation_data` JSON, with legacy records treated as Rust-bound at revision zero; Pi uses the existing message `request_id` as its `run_id`. User-visible messages, Resource Snapshots, and normal typed tool activity remain in the existing `agent_tasks` protobuf blobs. Bridge-only recovery data uses two Local-only sidecar tables: `agent_runtime_runs` stores each Agent Run Record, and `agent_tool_execution_records` stores request fingerprints, execution state, complete outcomes, and fixed Tool Result Projections under the composite tool identity. These tables are control data and never become a second Conversation Record.

Warp inserts a pending Tool Execution Record before permission or execution and atomically updates it after the tool returns. If recovery finds a record that entered execution but lacks a durable outcome, it becomes an Indeterminate Tool Execution and is never automatically re-executed; safety takes precedence over manufacturing a result. Deleting a Conversation Record deletes both sidecar sets in the same transaction. Because this adds database migrations and changes a crash-recovery boundary, implementation requires the repository's high-risk change plan and focused migration, deletion, crash-window, and backward-compatibility review before code changes begin.

An Indeterminate Tool Execution is materialized into the Conversation Record as a Tool Outcome whose persisted Tool Result Projection has status `error`, code `tool_outcome_unknown`, and `may_have_executed: true`; it contains no invented output or success data. Warp shows the tool activity as outcome unknown and ends the current Agent Run as failed. The agent loop cannot continue automatically. A user may create a Retry Run, in which Pi receives the unknown-outcome projection and is instructed to inspect actual state before attempting another mutation. The first increment does not allow either the user or Pi to rewrite the immutable record as success or ordinary failure.

Tool execution follows a durable state sequence. Warp first inserts the pending Tool Execution Record; after approval it durably marks the record `executing`; only then does it invoke the effect. When the tool returns, one SQLite transaction stores the complete outcome and fixed Tool Result Projection, updates the corresponding `agent_tasks` message, and advances the Conversation Record Revision. Warp returns the result to Pi and crosses the Commit Barrier only after that transaction succeeds. If the transaction fails after the effect, the durable state remains `executing` and recovery produces an Indeterminate Tool Execution. UI progress may be optimistic, but no later model turn or tool effect may depend on an uncommitted result.

This durable acknowledgement is added through a Pi-specific `CommitAgentRuntimeMutation` command to the existing SQLite writer rather than changing every Conversation persistence call. The command carries a stable `commit_id`, expected Conversation Record Revision, updated task data, optional sidecar mutation, and a one-shot response channel. Every Pi-bound change that affects the Runtime Transcript, including user input, completed assistant messages, tool activity, and history edits, uses this command. The writer applies the mutation with a revision compare-and-swap inside one transaction and returns the new revision. A revision conflict cancels the Agent Run, terminates that conversation's Pi process, and forces complete reconstruction. Re-delivering the same `commit_id` after an acknowledgement loss returns the already committed revision without duplicating data. Rust-bound conversations retain the existing fire-and-forget `UpdateMultiAgentConversation` path.

The first increment executes tool requests sequentially, including requests emitted together by one model turn. Parallel execution is disabled until a later Bridge Protocol capability defines ordering, approval presentation, cancellation, and idempotent recovery for concurrent tool effects.

Context Compaction is internal to Pi and may be stored in the Runtime Checkpoint, while Warp retains the complete Conversation Record. Compaction status may be surfaced to the UI, but summaries are not inserted as user-visible messages or used to rewrite history. If the checkpoint is lost, Pi derives new bounded context from the complete record.

The first Bridge Protocol event surface is limited to text deltas, completed-message commit requests, tool requests, run status, and terminal completed, cancelled, failed, or `limit_reached` outcomes. It does not expose raw reasoning or thinking content, and Provider-specific usage, cost, cache, and internal Pi event metadata require later negotiated capabilities. The protocol models stable Warp integration needs rather than mirroring Pi's internal event union.

Bridge failures use a fixed, typed, and redacted error taxonomy; Warp owns user-visible error copy and presentation. Provider response bodies, stack traces, API Keys, unredacted endpoint details, and Pi internal error objects never cross the protocol. Unknown failures map to a stable runtime-failure category with a non-sensitive diagnostic identity rather than raw error text.

Diagnostics are content-free by default. Bridge stdout is reserved for Bridge Protocol JSONL; stderr may contain only bounded, redacted metadata such as versions, run and diagnostic identities, state transitions, error categories, durations, retry or message counts, and process exit status. Prompts, model output, Conversation Record content, tool arguments or results, commands, file content, credentials, raw Provider responses, and unredacted endpoint details are never logged.

Runtime Checkpoints are in-memory only in the first increment. Pi session persistence is disabled, the Bridge writes no independent conversation transcript or compaction file, and process or application restart rebuilds runtime state from Warp's Conversation Record. A durable checkpoint cache requires a later decision justified by measured reconstruction cost.

Bridge absence, handshake failure, incompatibility, or startup failure does not silently fall back to Warp's Rust direct-provider runtime. Runtime selection occurs explicitly before a Conversation Record starts using it, and one conversation never mixes implementations. A Feature Flag may retain the Rust runtime as an overall rollout and rollback path until the Pi integration stabilizes.

The first increment applies only to Interactive Agent Conversations initiated through Warp's GUI or TUI. Passive suggestions are disabled when the Pi Agent Runtime is selected because their out-of-band, sometimes ephemeral lifecycle has no canonical Conversation Record or dedicated runtime process. The `warp agent` CLI and Agent SDK, including Ambient and Cloud Agent runs, retain their existing AgentDriver and harness paths and do not use the Bridge Protocol.

Every Conversation Record has an immutable, persisted Runtime Binding. Legacy records without a binding are treated as owned by Warp's Rust direct-provider runtime, while only newly created conversations may bind to Pi when the Feature Flag selects it. Changing the flag never migrates an existing conversation or mixes runtimes within it. If a Pi-bound conversation cannot start its Bridge, Warp keeps the record viewable but does not continue it through Rust; the user may explicitly fork its completed history into a new Rust-bound conversation.

Warp reconstructs Pi state by projecting the canonical Conversation Record into a versioned Runtime Transcript. The projection contains completed user and assistant messages, their Resource Snapshots, and paired tool requests and results; it excludes UI-only state, transient events, and Interrupted Output. Warp sends this stable protocol representation rather than database rows or Rust persistence types, and the Bridge validates and converts it into Pi messages without reading Warp's database. Each transcript carries a Conversation Record revision that Pi must acknowledge before Warp starts an Agent Run against it.

The first increment preserves Warp's existing behavior for user input submitted during an active Agent Run and does not expose Pi steering or follow-up queues. A direct submission cancels the active run before starting a new Agent Run. A Queued Prompt remains owned by Warp and starts a new run only after the active run reaches a terminal outcome; it is never injected into that run. The initial Bridge command surface therefore needs run start and cancellation, not mid-run steering.

Complete assistant messages are Commit Barriers. Text deltas remain unacknowledged display events, but each completed assistant message has a stable event identity and the Bridge waits for Warp to confirm durable incorporation into the Conversation Record through the same revisioned `CommitAgentRuntimeMutation` path. Re-delivery of the same event identity is idempotent. Pi must not request a tool, start another model turn, or complete the Agent Run before that confirmation. Warp likewise records a Tool Request before approval or execution and persists its Tool Outcome before returning it to Pi. A commit timeout fails the run without allowing later tool effects.

Any Conversation Record mutation that changes its Runtime Transcript causes Checkpoint Invalidation. Warp first cancels an active Agent Run, terminates that conversation's Pi process, and on the next run starts a fresh process with the complete new transcript. The first increment has no incremental history insertion or deletion commands. A fork always creates a new Conversation Record and dedicated process, inherits the source Runtime Binding by default, and changes runtime only when the user explicitly requests a runtime-conversion fork.

Each Agent Run receives one immutable Run Configuration assembled by Warp at start. It contains the OpenAI-compatible Provider Base URL, Model, and API Key; Tool Catalog schemas; selected Agent Resource Catalog content; working directory; context limit; and reasoning settings. Configuration changes during a run affect only later runs, and the Bridge receives no piecemeal settings updates. Tool permission policy is deliberately excluded from the snapshot: the Tool Execution Authority evaluates the current Warp policy for every Tool Request.

The first Bridge Protocol version allows only bounded text and image Runtime Content Blocks for model-visible messages and Tool Outcomes. Warp reads and normalizes text files, selections, diffs, and project context before sending them; images carry base64 data and a MIME type under Warp's attachment limits. Arbitrary binary attachments, URLs, and local path references are rejected before a run starts, and the Bridge never opens a path supplied as message content. Unsupported content produces an explicit error rather than being dropped or flattened silently.

Runtime Transcripts use a bounded, transactional Transcript Sync over the JSONL Bridge Protocol rather than one unbounded frame. Warp sends a sync-begin record with the conversation identity, revision, and item count; individual message or resource items; then a sync-commit record. The Bridge buffers and validates the candidate transcript, atomically replaces runtime context only at commit, and acknowledges the accepted revision before a run may start. Negotiated per-frame and total-sync limits are enforced, and interruption or validation failure discards the entire candidate sync.

The first increment narrows OpenAI-compatible Provider support to the Chat Completions protocol already used by Warp. Base URLs retain Warp's existing `/chat/completions` normalization, and Pi uses its `openai-completions` adapter with streaming output and function tools. The Bridge neither probes for nor falls back to the Responses API or another Provider protocol. Unsupported protocol configuration fails explicitly; adding another protocol later requires an explicit Provider configuration field and negotiated Bridge capability.

Warp retains each complete, typed Tool Outcome in its Tool Execution Record and also creates one bounded Tool Result Projection for model context. The projection contains a stable success, error, or denied status; text or image Runtime Content Blocks; and an explicit truncation marker when needed. It is persisted with the execution record and reused byte-for-byte for recovery or repeated delivery instead of being regenerated under newer code. Warp owns tool-specific truncation while retaining the complete result for UI and recovery; the Bridge never interprets Warp result types, stringifies them generically, or performs hidden truncation or summarization.

An app-wide Runtime Supervisor owns Bridge processes and indexes them by Conversation Record identity. View-scoped GUI and TUI controllers submit commands and consume events but never own the child process. The supervisor enforces at most one process and one active Agent Run per conversation across all surfaces, so moving or reopening a conversation attaches to the same runtime rather than duplicating execution. Closing a view does not immediately kill its process; deletion, Checkpoint Invalidation, and application exit force cleanup, while a centralized idle policy may evict an inactive process that can later be rebuilt from its Runtime Transcript.

Runtime selection for the first increment is rollout policy rather than a general user setting. Only new Interactive Agent Conversations in Local-only Mode may bind to Pi, and they do so only when the Pi Agent Runtime Feature Flag is enabled and the Chat Completions Provider configuration is valid. With the flag disabled, new Local-only conversations retain the Rust direct-provider binding; Account, Anonymous-only, Ambient and Cloud, and CLI paths remain unchanged. An existing Runtime Binding always overrides current flag state. A Pi-bound conversation that cannot continue may offer the previously defined explicit fork into the legacy runtime, but Warp does not expose a persistent Pi-versus-Rust selector.

The `warp-bridge` package owns a versioned Agent Policy Prompt tailored to this integration. It does not reuse Pi's default coding-agent prompt, which advertises Pi built-in tools, documentation, and discovery behavior that are disabled here. The Bridge renders its prompt from the structured Tool Catalog, Agent Resource Catalog, working directory, and date supplied in the Run Configuration; Warp does not send an arbitrary complete system-prompt blob. Permission guarantees remain enforced by Warp rather than prompt text. The prompt version is available as handshake and diagnostic metadata, while prompt content remains excluded from logs.
